## 1. 시스템 개요

MeeTeam의 PR 리뷰 시스템은 **GitHub Webhook**을 통해 PR 이벤트를 수신하고, **GPT-4o-mini**를 활용하여 자동 코드 리뷰를 수행하는 시스템입니다.

### 핵심 특징
- **이벤트 기반**: GitHub Webhook으로 PR 생성/수정 감지
- **비동기 처리**: @Async + CompletableFuture로 논블로킹 처리
- **병렬 리뷰**: 파일별 동시 LLM 호출 (Semaphore로 5개 제한)
- **AI 기반**: Spring AI + OpenAI GPT-4o-mini 연동

### 4.3 왜 이렇게 나눴는가?

```
┌─────────────────────────────────────────────────────────────────────┐
│  동기로 처리하는 이유 (#1, #2)                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  • PR 데이터가 없으면 리뷰 자체가 불가능                            │

│  • 실패 시 즉시 에러 응답 필요                                      │
│  • 상대적으로 빠름 (수백 ms ~ 수초)                                 │
│                                                                     │

┌─────────────────────────────────────────────────────────────────────┐
│  비동기로 처리하는 이유 (#3, #4)                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  • LLM 호출은 매우 느림 (파일당 2~10초)                             │

│  • 병렬 처리로 전체 시간 단축 가능                                  │
```

### 5.2 트랜잭션 전파 전략

```java
// 1. Facade 레이어 - 트랜잭션 없이 시작
@Transactional(propagation = Propagation.NEVER)
public PrReviewJob startPrReview(PullRequest pr, Long memberId) {
    // Job 생성은 새 트랜잭션
    PrReviewJob job = createOrReuseReviewJobInTransaction(pr, memberId);

    // 비동기 실행 시작 (즉시 반환)
    orchestrationService.startReviewAsync(job.getId());

    return job;
}

// 2. Job 생성 - 별도 트랜잭션 (다른 스레드에서 조회 가능하게)
@Transactional(propagation = Propagation.REQUIRES_NEW)
protected PrReviewJob createOrReuseReviewJobInTransaction(...) {
    // ...
    prReviewJobRepository.flush(); // 즉시 DB 반영
}

// 3. 비동기 실행 - 트랜잭션 없이 (내부에서 개별 관리)
@Async
@Transactional(propagation = Propagation.NEVER)
public CompletableFuture<Void> startReviewAsync(Long reviewJobId) {
    // 상태 업데이트는 별도 트랜잭션
    prReviewJobService.updateReviewJobStatus(reviewJobId, RUNNING);

    // LLM 오케스트레이션
    return llmOrchestrator.startPrReview(reviewJobId);
}
```

### 5.3 CompletableFuture 체인

```java
// LlmOrchestratorImpl.orchestratePrReview()

// 1. 파일별 CompletableFuture 생성 (병렬 실행)

    });

// 4. 요약 완료 대기
```

## 6. 동시성 제어

### 6.1 Semaphore 기반 제어

```
┌─────────────────────────────────────────────────────────────────────┐
│                    LLM 동시성 제어 (Semaphore)                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  문제: OpenAI API Rate Limit + 서버 리소스 보호                     │
│  해결: Semaphore(5)로 동시 호출 5개 제한                            │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐    │
│  │  Semaphore permits: 5                                       │    │
│  │                                                             │    │
│  │  Thread-1: acquire() ✅ → LLM 호출 → release()              │    │
│  │  Thread-2: acquire() ✅ → LLM 호출 → release()              │    │
│  │  Thread-3: acquire() ✅ → LLM 호출 → release()              │    │
│  │  Thread-4: acquire() ✅ → LLM 호출 → release()              │    │
│  │  Thread-5: acquire() ✅ → LLM 호출 → release()              │    │
│  │  Thread-6: acquire() ⏳ 대기중...                           │    │
│  │  Thread-7: acquire() ⏳ 대기중...                           │    │
│  │                                                             │    │
│  │  Thread-1 release() → Thread-6 acquire() ✅                 │    │
│  └─────────────────────────────────────────────────────────────┘    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 6.2 코드 구현

```java
// LlmConcurrencyLimiter.java
@Component
public class LlmConcurrencyLimiter {
    private final Semaphore semaphore = new Semaphore(5);

    public void acquire() throws InterruptedException {
        semaphore.acquire();
    }

    public void release() {
        semaphore.release();
    }
}

// LlmOrchestratorImpl.executeFileReviewAsync()
private CompletableFuture<LlmTaskResult> executeFileReviewAsync(LlmTask task) {
    return CompletableFuture.supplyAsync(() -> {
        try {
            limiter.acquire();  // 🔒 Semaphore 획득 (대기 가능)
            try {
                return llmReviewService.reviewFile(task);  // LLM 호출
            } finally {
                limiter.release();  // 🔓 반드시 해제
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException(e);
        }
    }, asyncTaskExecutor);
}
```

### 6.3 스레드 풀 설정

| 풀 이름 | 용도 | Core | Max | Queue | Keep-Alive |
|--------|------|------|-----|-------|------------|
| **async-thread** | @Async 기본 | 3 | 10 | 50 | - |
| **LlmTask** | CompletableFuture | 0 | 500 | 0 | 60초 |

```java
// AsyncConfig.java

// @Async 기본 Executor
@Override
public Executor getAsyncExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(3);
    executor.setMaxPoolSize(10);
    executor.setQueueCapacity(50);
    executor.setThreadNamePrefix("async-thread-");
    return executor;
}

// LLM 작업 전용 Executor
@Bean(name = "asyncTaskExecutor")
public Executor asyncTaskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(0);       // 필요시 생성
    executor.setMaxPoolSize(500);      // 최대 500개
    executor.setQueueCapacity(0);      // 큐 없음 (직접 스레드 생성)
    executor.setKeepAliveSeconds(60);  // 유휴 스레드 60초 후 제거
    executor.setThreadNamePrefix("LlmTask-");
    return executor;
}
```

## 8. 면접 대비

### 8.1 "PR 리뷰 시스템의 전체 흐름을 설명해주세요"

> "GitHub에서 PR이 생성되면 Webhook으로 이벤트를 수신합니다.
> 먼저 **동기적으로** PR 정보와 파일 목록을 GitHub API에서 가져와 DB에 저장합니다.
>
> 그 다음 **@Async로 별도 스레드**에서 코드 리뷰를 시작합니다.
> 파일별로 **CompletableFuture**를 생성하여 **병렬로 GPT-4o-mini**를 호출하고,
> **Semaphore(5)**로 동시 호출 수를 제한합니다.
>
> 모든 파일 리뷰가 완료되면 **allOf()**로 대기 후 전체 요약을 생성합니다."

### 8.2 "왜 비동기로 처리했나요?"

> "LLM API 호출은 파일당 2~10초가 걸리는 **I/O Bound 작업**입니다.
> 파일이 10개면 순차 처리 시 최대 100초가 걸릴 수 있고,
> GitHub Webhook은 30초 타임아웃이 있어서 동기 처리가 불가능합니다.
>
> 비동기 처리로:
> 1. **Webhook 응답을 즉시 반환** (GitHub 타임아웃 회피)
> 2. **병렬 처리로 전체 시간 단축** (100초 → 20초)
> 3. **실패 시 Job 상태로 재시도 가능**"

### 8.3 "I/O Bound 작업이 어디서 발생하나요?"

> "총 4곳에서 발생합니다:
>
> **동기 구간:**
> 1. GitHub App Token 발급 (~500ms)
> 2. PR 상세 정보 조회 (~800ms)
> 3. PR 파일 목록 조회 (~3초, 최대 300파일)
>
> **비동기 구간:**
> 4. 파일별 LLM 리뷰 (2~10초/파일, 병렬)
> 5. PR 전체 요약 (2~5초)
>
> 동기 구간은 데이터 수집이라 필수 선행 작업이고,
> 비동기 구간은 오래 걸리므로 병렬 처리합니다."

### 8.4 "Semaphore를 왜 5개로 제한했나요?"

> "두 가지 이유가 있습니다:
>
> 1. **OpenAI API Rate Limit**: 분당 요청 수 제한이 있어서
>    너무 많이 동시 호출하면 429 에러가 발생합니다.
>
> 2. **서버 리소스 보호**: 무제한 병렬 처리 시 메모리/커넥션 부족 가능.
>    5개는 경험적으로 안정적인 수치입니다.
>
> 5개로도 순차 처리 대비 5배 빠르고, Rate Limit을 회피할 수 있습니다."

### 8.5 "트랜잭션은 어떻게 관리하나요?"

> "레이어별로 다르게 관리합니다:
>
> 1. **Facade 레이어** (`PrReviewService`): `PROPAGATION_NEVER`
>    - 트랜잭션 없이 시작하여 각 작업이 독립적으로 커밋
>
> 2. **Job 생성** (`PrReviewJobService`): `REQUIRES_NEW`
>    - 별도 트랜잭션에서 즉시 커밋 + flush()
>    - 비동기 스레드에서 조회 가능하게
>
> 3. **비동기 실행** (`startReviewAsync`): `PROPAGATION_NEVER`
>    - 각 상태 업데이트를 개별 트랜잭션으로 처리
>    - 부분 실패 허용 (일부 파일만 성공 가능)"

### 9.1 ThreadPoolExecutor 동작 원리 관련

#### Q1. "ThreadPoolExecutor의 동작 원리를 직접 분석했다고 했는데, 어떻게 분석했나요?"

```
예상 꼬리질문:
• 소스코드를 직접 봤나요, 문서를 봤나요?
• execute() 메서드 내부 동작을 설명해줄 수 있나요?

답변 포인트:
1. JDK 소스코드 직접 분석 (ThreadPoolExecutor.execute())
2. 작업 제출 시 동작 순서:
   ① corePoolSize 미만 → 새 스레드 생성
   ② corePoolSize 이상 → 큐에 적재 시도
   ③ 큐가 가득 참 → maxPoolSize까지 스레드 생성
   ④ maxPoolSize 초과 → RejectedExecutionHandler 실행
```

---

#### Q2. "고정 스레드풀에서 '큐가 가득 차야 초과 스레드가 생성된다'고 했는데, 정확히 설명해주세요."

```
답변 포인트:

FixedThreadPool 내부 구조:
• corePoolSize == maxPoolSize (예: 둘 다 10)
• LinkedBlockingQueue (무제한 큐)

문제점:
• 큐가 무제한이므로 절대 "가득 차지" 않음
• 따라서 초과 스레드가 절대 생성되지 않음
• 10개 스레드가 바쁘면 나머지 작업은 무조건 큐 대기

실제 경험:
• PR 파일 8개 → 스레드 10개면 충분할 것 같지만
• GPT API 호출이 느려서 스레드가 오래 점유
• 새 PR 요청이 들어오면 큐에 계속 쌓임 → 지연 누적
```

---

#### Q3. "그러면 maxPoolSize를 늘리면 해결되는 거 아닌가요? 왜 CachedThreadPool로 바꿨나요?"

```
답변 포인트:

maxPoolSize만 늘리면 안 되는 이유:
• LinkedBlockingQueue가 무제한이므로 큐가 가득 차지 않음
• maxPoolSize를 100으로 해도 실제로는 corePoolSize만 사용
• 큐가 가득 차야 초과 스레드가 생성되는 구조

CachedThreadPool 선택 이유:
• SynchronousQueue 사용 (용량 0)
• 큐에 넣기 실패 → 즉시 새 스레드 생성
• 트래픽 증가 시점에 즉각적인 확장 가능
```

---

### 9.2 SynchronousQueue 관련

#### Q4. "SynchronousQueue가 뭔가요? 왜 용량이 0인 큐를 사용하나요?"

```
답변 포인트:

SynchronousQueue 특징:
• 저장 공간이 없는 특수 큐
• 생산자가 offer() 하면 소비자가 즉시 take() 해야 함
• 소비자가 없으면 offer() 실패 (false 반환)

ThreadPoolExecutor에서의 동작:
• 작업 offer() → 유휴 스레드가 있으면 즉시 전달 (handoff)
• 유휴 스레드 없으면 offer() 실패 → 새 스레드 생성

왜 사용했나:
• 큐 대기 시간 제거 (0ms)
• 작업 즉시 처리 보장
• PR 리뷰처럼 즉각적인 응답이 필요한 경우 적합
```

---

#### Q5. "SynchronousQueue에서 모든 스레드가 바쁘면 어떻게 되나요?"

```
답변 포인트:

CachedThreadPool의 경우:
• maxPoolSize = Integer.MAX_VALUE
• 새 스레드 생성 (사실상 무제한)

위험성 인지:
• 요청 폭증 시 스레드 무한 생성 → OOM 가능
• 스레드당 1MB 스택 메모리 사용

우리의 대응:
• Semaphore(5)로 GPT API 동시 호출 제한
• CPU 70% 모니터링 + scale out 계획
```

### 9.3 성능 측정 관련

#### Q6. "8초 → 2초로 단축됐다고 했는데, 어떻게 측정했나요?"

```
답변 포인트:

테스트 환경:
• Docker Compose, CPU 4코어 제한
• PR 파일 8개 고정

측정 방법:
• API 응답 시간 직접 측정 (Postman 또는 로그)
• 개선 전/후 동일 조건에서 여러 번 측정

계산 근거:
• 개선 전: 8개 파일 × ~1초/파일 = 8초 (순차적 처리에 가까움)
• 개선 후: 8개 파일 병렬 처리 → ~2초 (즉시 스레드 확장)
```

#### Q7. "왜 8초가 걸렸던 건가요? 비동기였는데 왜 순차처럼 동작했나요?"

```
GPT API 특성:
• 파일당 1~2초 소요 (I/O 대기)
• 스레드가 응답 기다리는 동안 점유 상태
• 새 작업은 큐에서 대기 → 지연 누적

결과:
• 병렬 처리가 아닌 준순차 처리처럼 동작
• 8개 파일이 차례로 처리되는 현상
```

```
위험성 인지:

우리의 대응:

   • CPU 70% 초과 시 scale out/up
   • Prometheus + Grafana 대시보드

3. keepAliveTime 활용
   • 60초 유휴 시 스레드 자동 제거
```

### 9.5 Virtual Thread 관련

#### Q10. "Virtual Thread를 도입했다고 했는데, 왜 도입했나요?"

```
답변 포인트:

문제 발견:
• 로그/모니터링으로 스레드 상태 분석
• 대부분 I/O 대기 상태 (GPT API 응답 대기)
• CPU는 거의 사용 안 하면서 스레드만 점유

Virtual Thread 효과:
• I/O 대기 시 캐리어 스레드 반환
• 적은 OS 스레드로 많은 작업 처리
• 스레드 생성 비용 대폭 감소 (1MB → 수백 바이트)
```

---

#### Q11. "Virtual Thread가 뭔지 설명해주세요. Platform Thread와 뭐가 다른가요?"

```
답변 포인트:

Platform Thread (기존):
• OS 스레드와 1:1 매핑
• 생성 비용 높음 (~1MB 스택)
• I/O 대기 중에도 OS 스레드 점유

Virtual Thread (Java 21+):
• JVM이 관리하는 경량 스레드
• 캐리어 스레드 위에서 실행
• I/O 대기 시 캐리어 스레드 반환 (unmount)
• 수백만 개 생성 가능

PR 리뷰에 적합한 이유:
• GPT API 호출 = 전형적인 I/O-bound 작업
• 대기 시간 동안 다른 작업 처리 가능
• 리소스 효율성 향상
```

---

#### Q12. "Virtual Thread를 사용하면 CachedThreadPool이 필요 없는 거 아닌가요?"

```
답변 포인트:

둘의 역할이 다름:
• CachedThreadPool: 작업 스케줄링 전략 (즉시 확장)
• Virtual Thread: 스레드 구현 방식 (경량화)

함께 사용 시 효과:
• CachedThreadPool → 요청 즉시 처리 보장
• Virtual Thread → I/O 대기 중 리소스 효율화
• 두 가지 최적화가 함께 동작

실제 적용:
• Executors.newVirtualThreadPerTaskExecutor() 사용
• 또는 CachedThreadPool + Virtual Thread 조합
```

### 9.7 깊이 있는 후속 질문

#### Q15. "Semaphore를 사용했다고 했는데, 몇 개로 제한했고 왜 그 숫자인가요?"

```
답변 포인트:

5개로 제한한 이유:
1. OpenAI API Rate Limit 고려
2. 비용 제어 (동시 요청 제한)
3. 테스트로 최적값 도출

Semaphore 동작:
• acquire() - 토큰 획득 (없으면 대기)
• release() - 토큰 반환
• GPT API 호출 전후로 사용
```

---

#### Q16. "만약 이 시스템에 요청이 폭주하면 어떻게 되나요?"

```
답변 포인트:

현재 구조의 동작:
1. CachedThreadPool → 스레드 계속 생성
2. Semaphore(5) → GPT 호출은 5개씩만
3. 나머지 스레드는 Semaphore 대기

잠재적 문제:
• Semaphore 대기 스레드가 많아지면 메모리 부담
• 대기 시간 증가 → 타임아웃 발생 가능

대응 계획:
• CPU 70% 모니터링 → scale out
• 타임아웃 설정으로 무한 대기 방지
• Circuit Breaker 도입 검토
```

---

#### Q17. "scale out과 scale up 중 어떤 걸 먼저 하실 건가요? 왜요?"

```
답변 포인트:

Scale out 우선 이유:
1. 비용 효율 (작은 인스턴스 여러 개)
2. 고가용성 (하나 죽어도 서비스 유지)
3. 점진적 확장 가능

Scale up 고려 시점:
• I/O보다 CPU-bound 작업이 많을 때
• 인스턴스 간 통신 오버헤드가 클 때
• 단일 요청 처리 성능이 중요할 때

PR 리뷰 시스템:
• I/O-bound → scale out이 효과적
• 각 인스턴스가 독립적으로 PR 처리 가능
```

## 10. Context Switching 이해

> **핵심 질문**: "I/O bound에서도 Context Switching이 많이 발생하는 거 아닌가요?"
>
> **답**: 맞습니다. 하지만 CPU bound와 **성격이 다릅니다**.

---

### 10.1 Context Switching 발생 원인

```
┌─────────────────────────────────────────────────────────────────┐
│  Context Switching이 발생하는 3가지 상황                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 타임 슬라이스 만료 (Preemptive) ← CPU bound 주 원인         │
│     ┌─────────────────────────────────────────────────────────┐ │
│     │ • OS가 강제로 CPU 뺏음 (보통 10~100ms마다)              │ │
│     │ • 스레드가 "일하고 있었는데" 강제 교체                  │ │
│     │ • 스레드 의지와 무관하게 발생                           │ │
│     └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│  2. I/O 대기 (Voluntary) ← I/O bound 주 원인                    │
│     ┌─────────────────────────────────────────────────────────┐ │
│     │ • 스레드가 자발적으로 CPU 양보                          │ │
│     │ • "나 기다려야 하니까 다른 애 먼저 해"                  │ │
│     │ • 네트워크, 디스크, API 응답 대기 시 발생               │ │
│     └─────────────────────────────────────────────────────────┘ │
│                                                                 │
│  3. 동기화 대기 (Blocking)                                      │
│     ┌─────────────────────────────────────────────────────────┐ │
│     │ • Lock, Semaphore, synchronized 획득 대기               │ │
│     │ • 다른 스레드가 자원 해제할 때까지 대기                 │ │
│     └─────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

---

### 10.2 CPU Bound vs I/O Bound Context Switching 비교

#### CPU Bound + 많은 스레드

```
┌─────────────────────────────────────────────────────────────────┐
│  CPU Bound 작업 (예: 이미지 처리, 암호화, 복잡한 계산)          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  시간 →                                                         │
│  ════════════════════════════════════════════════════════════   │
│                                                                 │
│  Thread-1: [계산중██████████] ─강제교체─▶ 대기                  │
│  Thread-2:        대기        [계산중██████████] ─강제교체─▶    │
│  Thread-3:                           대기         [계산중███    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 문제점                                                      ││
│  ├─────────────────────────────────────────────────────────────┤│
│  │ • 모든 스레드가 CPU를 "원함" (경쟁)                         ││
│  │ • 타임 슬라이스(10ms)마다 강제 교체 발생                    ││
│  │ • 교체 비용이 "순수 낭비"                                   ││
│  │   - 작업 중단됨                                             ││
│  │   - CPU 캐시 무효화 → 다시 로드해야 함                      ││
│  │   - 레지스터 저장/복원 비용                                 ││
│  │ • 스레드 많을수록 성능 저하                                 ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### I/O Bound + 많은 스레드 (PR 리뷰 시스템)

```
┌─────────────────────────────────────────────────────────────────┐
│  I/O Bound 작업 (예: GPT API 호출, DB 쿼리, 파일 읽기)          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  시간 →                                                         │
│  ════════════════════════════════════════════════════════════   │
│                                                                 │
│  Thread-1: [요청] ─양보─▶ ░░░░WAITING░░░░ ─응답─▶ [처리]        │
│  Thread-2:   [요청] ─양보─▶ ░░░░WAITING░░░░ ─응답─▶ [처리]      │
│  Thread-3:     [요청] ─양보─▶ ░░░░WAITING░░░░ ─응답─▶ [처리]    │
│                                                                 │
│            ↑         ↑                                          │
│         CPU 사용   CPU 미사용 (대기)                            │
│         (수 ms)    (1~2초)                                      │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 특징                                                        ││
│  ├─────────────────────────────────────────────────────────────┤│
│  │ • 스레드가 CPU를 "필요로 하지 않음"                         ││
│  │ • 자발적으로 양보 (어차피 기다려야 함)                      ││
│  │ • Context Switching 발생하지만 "낭비"가 아님                ││
│  │ • 다른 스레드가 CPU를 활용할 수 있음                        ││
│  │ • 스레드 많을수록 처리량 증가 (병렬 I/O)                    ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 10.3 핵심 차이: "낭비" vs "효율"

```
┌─────────────────────────────────────────────────────────────────┐
│                    Context Switching 비용 비교                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  CPU Bound:                                                     │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 교체 시점: 일하고 있었는데 강제로 뺏김                      ││
│  │ 교체 이유: 타임 슬라이스 만료 (OS 스케줄러)                 ││
│  │ 결과: 했던 작업 컨텍스트 저장 → 비용 "낭비"                 ││
│  │                                                             ││
│  │ [계산중...계산중...] ─강제교체─▶ (아깝다!)                  ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
│  I/O Bound:                                                     │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 교체 시점: 어차피 기다려야 해서 자발적 양보                 ││
│  │ 교체 이유: I/O 완료 대기 (스레드 스스로)                    ││
│  │ 결과: CPU 놀리지 않고 다른 작업 처리 → "효율적"             ││
│  │                                                             ││
│  │ [요청전송] ─자발적양보─▶ (어차피 기다려야 해)               ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```
│  GPT API 호출 시 Context Switching 흐름                         │

│  Thread-1                                                       │
│  ┌──────────────────────────────────────────────────────────┐   │

│  │         context switch                                   │   │
│  │         (자발적 양보)                                    │   │
│  └──────────────────────────────────────────────────────────┘   │
│                    ↓                                            │
│  Thread-2                                                       │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ [준비] → [GPT 요청 전송] → WAITING → [응답 처리] → ...   │   │
│  │              ↓                                           │   │
│  │         context switch                                   │   │
│  │         (자발적 양보)                                    │   │
│  └──────────────────────────────────────────────────────────┘   │
│                    ↓                                            │

│                                                                 │

│  │ 분석                                                        ││
│  ├─────────────────────────────────────────────────────────────┤│
│  │ • Context Switching 횟수: 많음 ✓                            ││
│  │ • 하지만 "낭비"인가?: 아니오 ✗                              ││
│  │ • 이유: 어차피 1~2초 기다려야 하는 시간에 다른 작업 처리    ││
│  │ • CPU 활용률: 오히려 높아짐 (놀지 않음)                     ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```
├─────────────────────────────────────────────────────────────────┤

│        ┌──────────┐                                             │

│                                                                 │
│  특징: RUNNABLE ↔ READY 반복 (타임슬라이스마다)                 │

│                                                                 │
│  특징: RUNNABLE → WAITING → RUNNABLE (I/O 완료 시)              │
```

### 10.6 비교 요약표

| 구분 | CPU Bound | I/O Bound (PR 리뷰) |
|------|-----------|---------------------|
| **CS 발생 빈도** | 많음 | 많음 |
| **CS 발생 원인** | 타임슬라이스 만료 (강제) | I/O 대기 (자발적) |
| **CS 발생 시점** | 작업 중 | 대기 필요 시 |
| **CS 비용 성격** | 순수 낭비 | 효율적 활용 |
| **스레드 많으면** | 성능 저하 | 처리량 증가 |
| **최적 스레드 수** | CPU 코어 수 | 코어 수보다 훨씬 많아도 OK |
| **CPU 사용률** | 100% (경쟁) | 낮음 (대기 시간 많음) |

### 10.7 결론

```
┌─────────────────────────────────────────────────────────────────┐
│  "Context Switching이 많다 = 무조건 나쁘다" 는 틀린 말          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  CPU Bound에서 CS가 많으면:                                     │
│  → 스레드들이 CPU를 놓고 경쟁                                   │
│  → 강제 교체로 인한 오버헤드 낭비                               │
│  → 스레드 수 줄이는 것이 해결책                                 │
│                                                                 │
│  I/O Bound에서 CS가 많으면:                                     │
│  → 스레드들이 자발적으로 양보                                   │
│  → CPU가 놀지 않고 다른 작업 처리                               │
│  → 오히려 효율적인 리소스 활용                                  │
│                                                                 │
│  PR 리뷰 시스템의 경우:                                         │
│  → GPT API 호출 = 전형적인 I/O Bound                            │
│  → CS가 많이 발생하지만 이건 "좋은 CS"                          │
│  → 스레드 늘려도 성능 저하 없음 (오히려 처리량 증가)            │
│  → Virtual Thread 도입으로 CS 비용 자체를 더 줄임               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 10.8 면접 예상 질문

#### Q. "I/O bound에서도 Context Switching이 발생하는데, 스레드를 많이 만들어도 괜찮은 이유가 뭔가요?"

```
답변 포인트:

1. CS 발생 원인이 다름
   • CPU bound: 타임슬라이스 만료 (강제)
   • I/O bound: I/O 대기 (자발적)

2. CS 비용의 성격이 다름
   • CPU bound: 작업 중 강제 중단 → 낭비
   • I/O bound: 어차피 대기 → 효율적 양보

3. PR 리뷰 시스템 예시
   • GPT API 응답 1~2초 대기
   • 이 시간 동안 CPU는 다른 스레드 처리
   • 스레드 많을수록 병렬 I/O 처리량 증가

4. 추가 최적화
   • Virtual Thread로 CS 비용 자체를 줄임
   • Platform Thread 수준의 CS → JVM 수준 mount/unmount
```
