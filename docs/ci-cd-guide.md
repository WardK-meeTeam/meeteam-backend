# CI/CD 쉽게 이해하기

## CI/CD가 없던 시절 (수동 배포)

개발자가 코드를 수정하고 서버에 반영하려면 이런 과정을 거쳤습니다:

```
1. 코드 수정 완료
2. 터미널 열기
3. ./gradlew build 실행
4. 빌드 성공할 때까지 기다림
5. FileZilla 열어서 서버 접속
6. JAR 파일 업로드
7. SSH로 서버 접속
8. 기존 프로세스 종료
9. 새 JAR 파일 실행
10. 로그 확인하며 정상 동작 확인
```

**문제점:**
- 매번 10단계를 수동으로 해야 함
- 실수하기 쉬움 (잘못된 파일 올림, 명령어 실수 등)
- 시간이 오래 걸림
- 테스트 깜빡하고 배포하면 버그 발생

> **CI/CD는 이 모든 과정을 자동화하는 것입니다.**

## CI란? (Continuous Integration)

### 한 줄 정의

> **"코드를 합칠 때마다 자동으로 빌드하고 테스트하는 것"**

### 비유: 레고 조립

```
팀원 3명이 각자 레고 부품을 만들고 있다고 상상해보세요.

❌ CI 없이 작업하면:
─────────────────────
철수: 빨간 블록 100개 완성!
영희: 파란 블록 100개 완성!
민수: 노란 블록 100개 완성!

→ 마지막에 합치려고 보니...
→ 철수 블록이랑 영희 블록이 안 맞음! 😱
→ 처음부터 다시 만들어야 함


✅ CI로 작업하면:
─────────────────────
철수: 빨간 블록 10개 완성 → [합쳐봄] → OK!
영희: 파란 블록 10개 완성 → [합쳐봄] → OK!
민수: 노란 블록 10개 완성 → [합쳐봄] → OK!
철수: 빨간 블록 10개 더 완성 → [합쳐봄] → 어? 안 맞네!
→ 바로 수정! (10개만 고치면 됨)

→ 자주 합치니까 문제를 빨리 발견!
```

### 실제 개발에서는?

```
              개발자 A가 push
                    │
                    ▼
    ┌───────────────────────────────┐
    │      CI 서버 (자동 실행)        │
    │                               │
    │  1. 코드 다운로드              │
    │  2. 빌드 (컴파일)              │
    │  3. 테스트 실행               │
    │  4. 코드 품질 검사             │
    │                               │
    │  결과: ✅ 성공 or ❌ 실패       │
    └───────────────────────────────┘
                    │
                    ▼
         ┌─────────────────┐
         │  결과 알림 전송   │
         │  (Slack, 이메일)  │
         └─────────────────┘
```

### CI가 하는 일

| 단계 | 설명 | 실패하면? |
|------|------|----------|
| **빌드** | 코드를 실행 가능한 형태로 변환 | 문법 오류, import 오류 발견 |
| **테스트** | 작성한 테스트 코드 실행 | 기능이 깨진 것 발견 |
| **린트/정적분석** | 코드 스타일, 잠재적 버그 검사 | 코드 품질 문제 발견 |

### CI의 핵심 가치

```
"문제를 빨리 발견하면, 고치기도 쉽다"

┌─────────────────────────────────────────────────────┐
│                                                      │
│   버그 발견 시점        vs       수정 비용           │
│                                                      │
│   코딩 직후 발견        →        5분이면 수정        │
│   1주일 후 발견         →        하루 걸림           │
│   배포 후 발견          →        며칠 + 야근 😭      │
│                                                      │
└─────────────────────────────────────────────────────┘

CI는 코딩 직후 바로 확인하게 해줍니다!
```

## CD란? (Continuous Delivery / Deployment)

### 두 가지 의미가 있어요

```
CD
├── Continuous Delivery (지속적 제공)
│   └── 배포 "준비"까지 자동화 (배포는 수동 버튼)
│
└── Continuous Deployment (지속적 배포)
    └── 배포까지 "전부" 자동화 (사람 개입 없음)
```

### 비유: 택배 시스템

```
온라인 쇼핑몰에서 주문했다고 상상해보세요.

┌─────────────────────────────────────────────────────┐
│                                                      │
│  [주문 접수] → [포장] → [배송] → [집 앞 도착]        │
│                                                      │
│  ════════════════════════════════════════════════   │
│                                                      │
│  Continuous Delivery (지속적 제공):                  │
│  ─────────────────────────────────                   │
│  [주문] → [포장] → [배송센터 도착] ──(대기)──→ [배송] │
│                          ↑                           │
│                    여기까지 자동                      │
│                    배송은 담당자가 버튼 클릭          │
│                                                      │
│                                                      │
│  Continuous Deployment (지속적 배포):                │
│  ───────────────────────────────────                 │
│  [주문] → [포장] → [배송센터] → [배송] → [도착!]     │
│                                                      │
│                    전부 자동! 사람 개입 없음          │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 실제 개발에서는?

```
                CI 성공!
                    │
                    ▼
    ┌───────────────────────────────┐
    │      CD 파이프라인 시작         │
    │                               │
    │  1. Docker 이미지 생성         │
    │  2. 이미지 저장소에 Push        │
    │  3. 서버에 접속                │
    │  4. 새 이미지 다운로드          │
    │  5. 서비스 재시작              │
    │                               │
    └───────────────────────────────┘
                    │
                    ▼
            🚀 배포 완료!
            사용자가 새 기능 사용 가능
```

## MeeTeam 프로젝트로 이해하기

### 우리 프로젝트의 CI/CD

```
[개발자가 master 브랜치에 Push]
              │
              ▼
┌─────────────────────────────────────────┐
│           GitHub Actions                 │
│                                          │
│  ╔══════════════════════════════════╗   │
│  ║            CI 단계                 ║   │
│  ╠══════════════════════════════════╣   │
│  ║  ✓ 코드 체크아웃                   ║   │
│  ║  ✓ JDK 17 설치                    ║   │
│  ║  ✓ Gradle 빌드                    ║   │
│  ║  ✗ 테스트 (현재 스킵됨 -x test)    ║   │
│  ╚══════════════════════════════════╝   │
│                  │                       │
│                  ▼                       │
│  ╔══════════════════════════════════╗   │
│  ║            CD 단계                 ║   │
│  ╠══════════════════════════════════╣   │
│  ║  ✓ Docker 이미지 생성              ║   │
│  ║  ✓ AWS ECR에 Push                 ║   │
│  ║  ✓ EC2 서버에 SSH 접속            ║   │
│  ║  ✓ 새 이미지 Pull & 실행           ║   │
│  ╚══════════════════════════════════╝   │
│                                          │
└─────────────────────────────────────────┘
              │
              ▼
      🚀 https://api.meeteam.alom-sejong.com
         새 버전 배포 완료!
```

### deploy.yml 파일과 매칭

```yaml
# .github/workflows/deploy.yml

# ═══════════════════════════════════════════════════════════
#                         CI 단계
# ═══════════════════════════════════════════════════════════

- name: Github Repository 파일 불러오기    # 코드 가져오기
  uses: actions/checkout@v4

- name: JDK 17버전 설치                    # 환경 설정
  uses: actions/setup-java@v4

- name: 테스트 및 빌드하기                  # 빌드 (테스트는 스킵)
  run: ./gradlew clean build -x test


# ═══════════════════════════════════════════════════════════
#                         CD 단계
# ═══════════════════════════════════════════════════════════

- name: Docker 이미지 생성                 # 도커 이미지 만들기
  run: docker build -t meeteam-server .

- name: ECR에 Docker 이미지 Push           # 이미지 저장소에 업로드
  run: docker push ... /meeteam-server:latest

- name: SSH 로 EC2 에 접속하기              # 서버에 배포
  uses: appleboy/ssh-action@v1.0.3
  script: |
    docker compose pull meeteam-server
    docker compose up -d meeteam-server
```

## 왜 CI/CD를 쓸까? (장점)

### 1. 자동화 = 실수 방지

```
수동 배포:
───────────────────────────────────
"어? jar 파일 잘못 올렸네"
"아 테스트 안 돌리고 배포했다"
"서버 주소 잘못 입력했네..."

CI/CD:
───────────────────────────────────
모든 단계가 스크립트로 정의됨
매번 똑같이 실행됨
실수할 여지가 없음
```

### 2. 빠른 피드백

```
┌─────────────────────────────────────────────────┐
│                                                  │
│   Push 후 5분 만에:                              │
│                                                  │
│   ✅ "빌드 성공했습니다"                          │
│   ❌ "테스트 3개 실패했습니다"                     │
│   ⚠️ "보안 취약점 발견: SQL Injection 위험"       │
│                                                  │
│   → 바로 문제를 알 수 있음!                       │
│                                                  │
└─────────────────────────────────────────────────┘
```

### 3. 배포 두려움 제거

```
CI/CD 없을 때:
───────────────────────────────────
"금요일 오후에 배포하면 주말에 장애나면 어쩌지..."
"배포할 때마다 긴장돼..."
"한 번에 많이 모아서 배포하자" → 문제 커짐

CI/CD 있을 때:
───────────────────────────────────
"자동으로 테스트 다 통과했으니 괜찮겠지"
"문제 생기면 바로 롤백하면 되지"
"작은 변경을 자주 배포" → 문제 작음
```

### 비교 표

| 항목 | 수동 배포 | CI/CD |
|------|----------|-------|
| 배포 시간 | 30분~1시간 | 5~10분 (자동) |
| 실수 가능성 | 높음 | 거의 없음 |
| 테스트 누락 | 자주 발생 | 불가능 (강제) |
| 배포 빈도 | 일주일에 1번 | 하루에 여러 번 |
| 롤백 | 복잡하고 오래 걸림 | 자동화 가능 |

## 면접 답변 예시

### Q: CI/CD가 뭔가요?

```
CI는 Continuous Integration으로, 개발자가 코드를 Push할 때마다
자동으로 빌드하고 테스트하여 문제를 빠르게 발견하는 것입니다.

CD는 Continuous Deployment로, CI를 통과한 코드를
자동으로 서버에 배포하는 것입니다.

저희 프로젝트에서는 GitHub Actions로 CI/CD를 구축했고,
master 브랜치에 Push하면 자동으로 빌드, Docker 이미지 생성,
ECR Push, EC2 배포까지 진행됩니다.
```

### Q: CI/CD의 장점은?

```
첫째, 자동화로 인한 실수 방지입니다.
수동 배포 시 발생할 수 있는 휴먼 에러를 제거합니다.

둘째, 빠른 피드백입니다.
코드 문제를 Push 직후 바로 알 수 있어서
수정 비용이 적습니다.

셋째, 배포 빈도를 높일 수 있습니다.
자동화되어 있으니 작은 변경을 자주 배포할 수 있고,
이는 문제 발생 시 원인 파악을 쉽게 합니다.
```
