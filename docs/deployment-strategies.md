# 배포 전략 가이드

## 왜 배포 전략이 중요할까?

서비스를 운영하다 보면 새로운 기능을 추가하거나 버그를 수정해야 합니다.
이때 **사용자에게 영향을 최소화**하면서 새 버전을 배포하는 방법이 바로 **배포 전략**입니다.

---

## 1. Rolling 배포 (롤링 배포)

### 개념

> "서버를 **하나씩 순차적으로** 교체하는 방식"

기존 서버들을 한 번에 모두 바꾸지 않고, **하나씩 차례대로** 새 버전으로 교체합니다.

### 동작 방식

```
[시작 상태] 서버 3대가 v1.0 운영 중

   ┌─────────┐  ┌─────────┐  ┌─────────┐
   │ Server1 │  │ Server2 │  │ Server3 │
   │  v1.0   │  │  v1.0   │  │  v1.0   │
   └─────────┘  └─────────┘  └─────────┘
        ▲            ▲            ▲
        └────────────┼────────────┘
                     │
              [Load Balancer]
                     │
                 [Users]


[1단계] Server1을 v2.0으로 교체

   ┌─────────┐  ┌─────────┐  ┌─────────┐
   │ Server1 │  │ Server2 │  │ Server3 │
   │  v2.0   │  │  v1.0   │  │  v1.0   │
   │   🔄    │  │         │  │         │
   └─────────┘  └─────────┘  └─────────┘

   ⚠️ 이 시점: v1.0과 v2.0이 동시에 서비스됨!


[2단계] Server2를 v2.0으로 교체

   ┌─────────┐  ┌─────────┐  ┌─────────┐
   │ Server1 │  │ Server2 │  │ Server3 │
   │  v2.0   │  │  v2.0   │  │  v1.0   │
   │         │  │   🔄    │  │         │
   └─────────┘  └─────────┘  └─────────┘


[3단계] Server3을 v2.0으로 교체 → 완료!

   ┌─────────┐  ┌─────────┐  ┌─────────┐
   │ Server1 │  │ Server2 │  │ Server3 │
   │  v2.0   │  │  v2.0   │  │  v2.0   │
   │   ✅    │  │   ✅    │  │   ✅    │
   └─────────┘  └─────────┘  └─────────┘
```

### 특징

| 항목 | 설명 |
|------|------|
| **서버 수** | 기존 서버 수 그대로 사용 |
| **배포 속도** | 서버 수에 비례 (느림) |
| **버전 혼재** | 배포 중 v1.0과 v2.0이 **동시에 존재** |
| **리소스** | 추가 서버 불필요 |
| **롤백** | 다시 역순으로 롤링 (느림) |

### 장점

- 추가 인프라 비용 없음
- 점진적 배포로 문제 조기 발견 가능
- 서버 리소스 효율적 사용

### 단점

- 배포 중 **두 버전이 공존** → 호환성 문제 가능
- 롤백 시간이 오래 걸림
- DB 스키마 변경 시 복잡함

### 현재 MeeTeam의 상황

```
MeeTeam은 서버가 1대이므로...

[v1.0 운영 중]        [v1.0 중지]         [v2.0 시작]
┌─────────┐          ┌─────────┐         ┌─────────┐
│ Server  │   ──▶    │ Server  │   ──▶   │ Server  │
│  v1.0   │          │   ⬛    │         │  v2.0   │
│  ✅     │          │ 다운타임 │         │   ✅    │
└─────────┘          └─────────┘         └─────────┘
     │                    │                   │
  서비스 중            서비스 불가           서비스 재개
                     (몇 초~몇 분)
```

**단일 서버 Rolling = 다운타임 발생!**

## 2. Blue/Green 배포 (블루/그린 배포)

### 개념

> "**동일한 환경 2개**를 준비해두고, 트래픽만 전환하는 방식"

- **Blue**: 현재 운영 중인 환경 (구버전)
- **Green**: 새로 배포할 환경 (신버전)

### 동작 방식

```
[시작 상태] Blue 환경이 v1.0 운영 중

                    [Load Balancer]
                          │
                          ▼
   ┌─────────────────────────────────────────────┐
   │                   BLUE                       │
   │  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
   │  │ Server1 │  │ Server2 │  │ Server3 │     │
   │  │  v1.0   │  │  v1.0   │  │  v1.0   │     │  ◀── 트래픽 100%
   │  └─────────┘  └─────────┘  └─────────┘     │
   └─────────────────────────────────────────────┘

   ┌─────────────────────────────────────────────┐
   │                   GREEN                      │
   │  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
   │  │ Server1 │  │ Server2 │  │ Server3 │     │
   │  │  (대기)  │  │  (대기)  │  │  (대기)  │     │  ◀── 트래픽 0%
   │  └─────────┘  └─────────┘  └─────────┘     │
   └─────────────────────────────────────────────┘


[1단계] Green 환경에 v2.0 배포 (사용자 영향 없음)

                    [Load Balancer]
                          │
                          ▼
   ┌─────────────────────────────────────────────┐
   │                   BLUE                       │
   │  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
   │  │  v1.0   │  │  v1.0   │  │  v1.0   │     │  ◀── 트래픽 100% (계속 서비스)
   │  └─────────┘  └─────────┘  └─────────┘     │
   └─────────────────────────────────────────────┘

   ┌─────────────────────────────────────────────┐
   │                   GREEN                      │
   │  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
   │  │  v2.0   │  │  v2.0   │  │  v2.0   │     │  ◀── 배포 & 테스트 중
   │  │   🔄    │  │   🔄    │  │   🔄    │     │
   │  └─────────┘  └─────────┘  └─────────┘     │
   └─────────────────────────────────────────────┘


[2단계] 트래픽 전환! (스위치처럼 즉시 전환)

                    [Load Balancer]
                          │
           ┌──────────────┴──────────────┐
           │                             │
           ▼                             ▼
   ┌───────────────────┐       ┌───────────────────┐
   │       BLUE        │       │       GREEN       │
   │  v1.0 (대기 전환)  │       │  v2.0 (서비스!)   │
   │     트래픽 0%     │       │    트래픽 100%    │
   └───────────────────┘       └───────────────────┘

   ✅ 전환 시간: 거의 즉시! (DNS/LB 설정 변경)
   ✅ 문제 발생 시: 다시 Blue로 전환 (즉시 롤백)
```

### 특징

| 항목 | 설명 |
|------|------|
| **서버 수** | 기존의 **2배** 필요 |
| **배포 속도** | 매우 빠름 (트래픽 전환만) |
| **버전 혼재** | **없음** (한 버전만 서비스) |
| **리소스** | 추가 인프라 비용 발생 |
| **롤백** | **즉시** (트래픽만 되돌림) |

### 장점

- **무중단 배포** (Zero Downtime)
- **즉시 롤백** 가능
- 배포 전 Green 환경에서 충분히 테스트 가능
- 버전 혼재 문제 없음

### 단점

- 인프라 비용 **2배**
- 데이터베이스 동기화 고려 필요
- 설정이 복잡함

## 3. 한눈에 비교

```
┌─────────────────────────────────────────────────────────────────┐
│                        배포 전략 비교                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   Rolling 배포                    Blue/Green 배포                │
│   ════════════                    ═══════════════                │
│                                                                  │
│   [v1] [v1] [v1]                  [  BLUE: v1  ] ◀── 서비스     │
│     │                             [  GREEN: -  ]                 │
│     ▼                                   │                        │
│   [v2] [v1] [v1]  ← 혼재!               ▼                        │
│     │                             [  BLUE: v1  ]                 │
│     ▼                             [  GREEN: v2 ] ← 배포          │
│   [v2] [v2] [v1]  ← 혼재!               │                        │
│     │                                   ▼ (스위치!)              │
│     ▼                             [  BLUE: v1  ]                 │
│   [v2] [v2] [v2]  ← 완료          [  GREEN: v2 ] ◀── 서비스     │
│                                                                  │
│   ⏱️ 느림                          ⏱️ 빠름                        │
│   💰 저렴                          💰 2배 비용                    │
│   🔄 롤백 느림                     🔄 롤백 즉시                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 상세 비교표

| 비교 항목 | Rolling 배포 | Blue/Green 배포 |
|-----------|-------------|-----------------|
| **다운타임** | 있을 수 있음 | 없음 (무중단) |
| **버전 혼재** | O (배포 중 공존) | X (한 버전만) |
| **롤백 속도** | 느림 (재배포) | 즉시 (트래픽 전환) |
| **인프라 비용** | 기존 그대로 | 2배 |
| **배포 복잡도** | 낮음 | 중간 |
| **테스트** | 프로덕션에서 바로 | 전환 전 테스트 가능 |
| **DB 마이그레이션** | 복잡 | 복잡 |

## 4. 그 외 배포 전략들

### Canary 배포 (카나리 배포)

> "새 버전을 **일부 사용자에게만** 먼저 배포"

```
[사용자 100명]
      │
      ├── 95명 ──▶ [v1.0] 기존 버전
      │
      └── 5명  ──▶ [v2.0] 새 버전 (테스트 그룹)

→ 문제 없으면 점진적으로 v2.0 비율 증가
→ 문제 발생하면 v2.0으로 가는 트래픽 차단
```

**장점**: 리스크 최소화, 실제 사용자로 테스트
**단점**: 복잡한 라우팅 설정 필요

### A/B 테스팅 배포

> "두 버전을 비교하여 **더 좋은 버전** 선택"

기능적 차이가 있는 두 버전을 동시에 운영하며 성과를 비교합니다.

## 5. MeeTeam에 적용한다면?

### 현재 상황

```
현재: 단일 EC2 + Docker Compose
      → docker stop → docker rm → docker up
      → 다운타임 발생 (수 초 ~ 수십 초)
```

### 개선 방안 1: Nginx를 활용한 Blue/Green

```
┌─────────────────────────────────────────────────────┐
│                    EC2 Instance                      │
│                                                      │
│  ┌──────────┐                                       │
│  │  Nginx   │ ◀── 트래픽 진입점                     │
│  │ (LB역할) │                                       │
│  └────┬─────┘                                       │
│       │                                              │
│       ├────────────────┬─────────────────┐          │
│       ▼                ▼                 ▼          │
│  ┌─────────┐     ┌─────────┐      ┌─────────┐      │
│  │  Blue   │     │  Green  │      │  Redis  │      │
│  │  :8080  │     │  :8081  │      │  :6379  │      │
│  │  v1.0   │     │  v2.0   │      │         │      │
│  └─────────┘     └─────────┘      └─────────┘      │
│                                                      │
└─────────────────────────────────────────────────────┘

배포 시:
1. Green(8081)에 새 버전 배포
2. 헬스체크 통과 확인
3. Nginx 설정 변경 (8080 → 8081)
4. nginx reload (무중단)
```

### 개선 방안 2: AWS ALB + Target Group

```
┌─────────────────────────────────────────────────────┐
│                       AWS                            │
│                                                      │
│  ┌───────────────────────────────┐                  │
│  │    Application Load Balancer  │                  │
│  └───────────────┬───────────────┘                  │
│                  │                                   │
│        ┌─────────┴─────────┐                        │
│        ▼                   ▼                        │
│  ┌───────────┐      ┌───────────┐                  │
│  │Target Grp │      │Target Grp │                  │
│  │   Blue    │      │   Green   │                  │
│  │  (v1.0)   │      │  (v2.0)   │                  │
│  └─────┬─────┘      └─────┬─────┘                  │
│        ▼                   ▼                        │
│  ┌───────────┐      ┌───────────┐                  │
│  │    EC2    │      │    EC2    │                  │
│  │   Blue    │      │   Green   │                  │
│  └───────────┘      └───────────┘                  │
│                                                      │
└─────────────────────────────────────────────────────┘

배포 시:
1. Green Target Group에 새 버전 배포
2. ALB 리스너 규칙 변경 (Blue → Green)
3. 즉시 트래픽 전환
```

## 6. 면접 질문 대비

### Q: Rolling 배포와 Blue/Green 배포의 차이는?

```
Rolling 배포는 기존 서버를 하나씩 순차적으로 교체하는 방식입니다.
추가 인프라 비용은 없지만, 배포 중 구버전과 신버전이 혼재하고
롤백 시 다시 순차적으로 되돌려야 합니다.

Blue/Green 배포는 동일한 환경 2개를 준비하고 트래픽만 전환하는 방식입니다.
인프라 비용이 2배지만, 무중단 배포와 즉시 롤백이 가능합니다.
```

### Q: 어떤 상황에서 어떤 전략을 선택하나요?

```
- 비용이 제한적이고 잠깐의 다운타임이 허용되면 → Rolling
- 무중단이 필수이고 빠른 롤백이 중요하면 → Blue/Green
- 리스크를 최소화하며 점진적 배포가 필요하면 → Canary
```

### Q: 현재 프로젝트의 배포 방식과 개선점은?

```
현재는 단일 EC2에서 컨테이너를 중지하고 새로 시작하는 방식이라
짧은 다운타임이 발생합니다.

개선한다면 같은 EC2 내에서 Nginx를 리버스 프록시로 두고
두 개의 컨테이너(Blue/Green)를 운영하여
트래픽 전환 방식으로 무중단 배포를 구현하겠습니다.
```
